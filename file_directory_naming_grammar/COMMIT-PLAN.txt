────────────────────────────────────────
COMMIT PLAN
────────────────────────────────────────

Artifact:
File & Directory Naming Grammar

Purpose:
Materialize an unambiguous, machine-executable naming grammar
derived from the completed discussion scaffold.

This plan is intended to be followed strictly.
Deviations are permitted only if surfaced by execution,
validation failure, or explicit inconsistency discovery.

────────────────────────────────────────
COMMIT 0 — Preconditions
────────────────────────────────────────

• Verify fresh GitHub access using repository shibboleth.
• Emit current NONCE and confirm live read.
• Abort if verification fails.

────────────────────────────────────────
COMMIT 1 — Canonical Specification Skeleton
────────────────────────────────────────

Objective:
Create the initial specification file with structure only.

Actions:
• Create new specification artifact file.
• Populate:
  – Title
  – Purpose
  – Scope
  – Normative Language Declaration (MUST / MUST NOT)
• Do NOT include rules yet.
• Header must conform to Deliverable Header Specification.

Acceptance Criteria:
• File exists in correct repository and path.
• Header is valid.
• Body contains no normative rules.

────────────────────────────────────────
COMMIT 2 — Scope & Applicability
────────────────────────────────────────

Objective:
Canonize Sections A, B, and I (Scope, Stability, Migration).

Actions:
• Convert discussion text into normative clauses.
• Remove all question prompts.
• Preserve intent exactly.
• Explicitly state:
  – Applies to files and directories equally
  – No legacy exemption
  – Immutability and rename semantics

Acceptance Criteria:
• No ambiguity remains about applicability.
• No references to discussion scaffolding remain.

────────────────────────────────────────
COMMIT 3 — Semantic Constraints
────────────────────────────────────────

Objective:
Canonize Section C (Forbidden Semantics).

Actions:
• Declare disallowed semantic classes normatively.
• Explicitly state that no other semantic restrictions apply.
• Ensure prohibition is phrased as MUST NOT.

Acceptance Criteria:
• Disallowed semantics are exhaustive and explicit.
• No prescriptive meaning guidance exists.

────────────────────────────────────────
COMMIT 4 — Character Set & Shape Grammar
────────────────────────────────────────

Objective:
Canonize Section D into enforceable grammar.

Actions:
• Declare character set, casing, and length constraints.
• Declare derived artifact naming pattern.
• Specify derivation rules and prohibitions.
• Optionally include formal grammar (BNF or equivalent).

Acceptance Criteria:
• Grammar is mechanically checkable.
• Validity of a name can be determined without interpretation.

────────────────────────────────────────
COMMIT 5 — Uniqueness & Collision Rules
────────────────────────────────────────

Objective:
Canonize Section E (Collisions).

Actions:
• Declare collision prohibition.
• Declare resolution semantics (merge, not suffix).
• Clarify precedence of existing artifacts.

Acceptance Criteria:
• No suffix-based resolution allowed.
• Collision handling is unambiguous.

────────────────────────────────────────
COMMIT 6 — Human vs Machine Interpretation
────────────────────────────────────────

Objective:
Canonize Section F.

Actions:
• Declare separation between human meaning and machine behavior.
• Explicitly forbid machine inference.
• Confirm ambiguity tolerance boundaries.

Acceptance Criteria:
• Machine behavior is fully constrained.
• Human intent has no machine authority.

────────────────────────────────────────
COMMIT 7 — Grammar vs Style & Ambiguity Handling
────────────────────────────────────────

Objective:
Canonize Section G.

Actions:
• Declare grammar as strictly normative.
• Eliminate any style layer.
• Declare ambiguity as spec failure.
• Declare loud failure and EMITTER notification.

Acceptance Criteria:
• No SHOULD language exists.
• Ambiguity handling is explicit and terminal.

────────────────────────────────────────
COMMIT 8 — Failure Mode & Enforcement
────────────────────────────────────────

Objective:
Canonize Section H.

Actions:
• Declare immediate rejection of nonconforming artifacts.
• Declare LLM behavior during header generation.
• Forbid partial or best-guess output.

Acceptance Criteria:
• Enforcement semantics are binary and immediate.
• LLMs are explicitly constrained.

────────────────────────────────────────
COMMIT 9 — Naming vs Placement Boundary
────────────────────────────────────────

Objective:
Canonize Section J.

Actions:
• Declare separation of grammar and placement.
• Declare no precedence.
• Declare conflict as failure.

Acceptance Criteria:
• No cross-layer inference permitted.
• Boundary is explicit.

────────────────────────────────────────
COMMIT 10 — Examples & Test Vectors (Non-Normative)
────────────────────────────────────────

Objective:
Add illustrative examples without softening rules.

Actions:
• Add valid name examples.
• Add invalid name examples with rejection reasons.
• Clearly mark examples as non-normative.

Acceptance Criteria:
• Examples do not introduce exceptions.
• Grammar remains sole authority.

────────────────────────────────────────
COMMIT 11 — Final Consistency Pass
────────────────────────────────────────

Objective:
Ensure internal consistency and closure.

Actions:
• Verify no contradictions across sections.
• Verify terminology consistency.
• Verify no reliance on memory or context.
• Run shibboleth verification before commit.

Acceptance Criteria:
• Spec is self-contained.
• Execution does not require external explanation.

────────────────────────────────────────
COMMIT 12 — Canonization
────────────────────────────────────────

Objective:
Declare the specification complete.

Actions:
• Final NONCE verification.
• Commit with clear message indicating canonization.
• Rotate NONCE post-commit.

Acceptance Criteria:
• Spec is authoritative.
• Further changes require new commit plan.
